# ADR-004: 조회수 업데이트 방식 및 N+1 문제 해결

## 상태
Accepted

## 컨텍스트
게시글 상세 조회 시 조회수(VIEW_COUNT)를 업데이트하는 효율적인 방법과, 목록 조회 시 연관 엔티티(User, Game)를 가져오는 과정에서 발생하는 N+1 문제를 해결해야 했습니다.

## 결정
1. **업데이트 방식**: 성능 비교 대조군(Baseline) 설정을 위해 우선 **DB 직접 업데이트** 방식을 구현했습니다.
2. **N+1 해결**: `JPQL fetch join`을 사용하여 연관 엔티티를 한 번의 쿼리로 묶어 조회합니다.

## 근거
- **Atomic Query**: `@Modifying`과 `@Query`를 조합하여 DB 레벨에서 바로 숫자를 올리는 원자적 쿼리(`update... set v = v + 1`)를 사용했습니다. 이는 단순 Dirty Checking보다 조회 후 수정 과정이 줄어들어 성능 저하를 완화합니다.
- **Fetch Join**: `Lazy Loading`으로 인해 게시글 수만큼 추가 쿼리가 발생하는 N+1 문제를 방지하여 쿼리 효율성을 극대화했습니다.

## 결과
- 단일 쿼리로 목록 조회가 가능해졌으며, 정합성이 보장된 조회수 카운팅이 가능합니다.
- (여담) DB 직접 업데이트 방식은 동시 요청 시 DB 락 경합이 예상되므로, 추후 Redis 캐싱 방식과의 성능 격차를 증명하는 지표로 활용할 예정입니다.
