# ADR-003: 조회수 업데이트 및 성능 최적화 (DB Update vs Redis)

## 상태
Accepted

## 컨텍스트
게시글 상세 조회 시 조회수(View Count)를 업데이트하는 로직이 필요했습니다. 또한, 목록 조회 시 연관 객체를 불러올 때 발생하는 성능 저하 문제를 해결해야 했습니다.

## 결정
1. **업데이트 방식**: 성능 비교 대조군 설정을 위해 우선 **DB 직접 업데이트** 방식을 구현했습니다.
2. **N+1 해결**: `fetch join`을 사용하여 연관 엔티티를 한 번에 조회하도록 처리했습니다.

## 근거
- **Atomic Query**: Dirty Checking(JPA 변경 감지)은 조회 후 수정하는 과정을 거치지만, `@Modifying`과 쿼리를 직접 사용한 `update p set p.viewCount = p.viewCount + 1` 방식은 DB 레벨에서 원자적 연산을 수행하여 동시성 문제를 완화합니다.
- **Fetch Join**: `Lazy Loading` 설정으로 인해 발생하는 N+1 문제를 해결하여 쿼리 횟수를 1/N 수준으로 단축했습니다.

## 결과
- 데이터의 정합성을 보장하면서 목록 조회 속도를 최적화했습니다.
- (여담) DB 직접 업데이트 방식은 동시 접속자가 많을 때 쓰기 지연(Write Contention)이 발생할 것으로 보이며, 이는 추후 Redis 캐싱과의 성능 비교 지표가 될 것입니다.
